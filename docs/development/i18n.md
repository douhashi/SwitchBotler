# 国際化（i18n）

UI 文言の多言語対応方針。react-i18next + i18next を基盤とし、翻訳リソースはアプリに同梱して同期ロードする。

## 基盤構成

- ライブラリ: `i18next` + `react-i18next`。
- 初期化: `src/i18n/index.ts` で `initReactI18next` を用いて init する（副作用 import）。`resources` を静的 import で同梱し、`fallbackLng: "en"` / `defaultNS: "common"` / `interpolation.escapeValue: false` / `react.useSuspense: false` を設定する。
- 言語解決: `src/i18n/resolve.ts` の `resolveLanguage(pref)` が、保存値（`ja` / `en`）優先 → `system` は `navigator.language` の先頭 2 文字で判定 → 未対応言語は `en` フォールバック、の順で描画言語を決める。
- 永続化: `src/stores/language-store.ts`（zustand `persist`、localStorage キー `switchbotler-language`）。`theme-store` と同型で、`setLanguage` は副作用（`applyLanguage` → `i18n.changeLanguage`）→ `set` の順に実行し、`onRehydrateStorage` で再適用する。依存方向は store → i18n の単方向（循環 import 回避）。
- ブートストラップ: `src/main.tsx` で `import "./i18n"` 後、描画前に `applyLanguage(useLanguageStore.getState().language)` を呼び、FOUC（初回ちらつき）を防ぐ。テーマ（`applyTheme`）と並置する。
- ウィンドウ間一貫: メイン / トレイは同一オリジンで localStorage を共有するため、言語設定も両ウィンドウで一貫する（plugin-store 不要）。

## 翻訳リソースの配置

```
src/i18n/locales/<lang>/<namespace>.json
```

- 対応言語: `ja` / `en`（SSoT は `src/i18n/resolve.ts` の `SUPPORTED_LANGUAGES`）。
- namespace: `common` / `settings` / `connection` / `onboarding` / `devices` / `scenes` / `sensors` / `tray` / `errors`（SSoT は `src/i18n/index.ts` の `namespaces`）。

## namespace 方針

- **`common`**: 画面横断の共通語彙。`defaultNS`。`actions.*`（save/cancel/retry/refresh/close/back）・`states.*`（loading/connected/disconnected/waiting/connectedCount）・`nav.*`（各画面名・aria）・`language.*`。
- **画面 / 機能名 namespace**: その画面・機能固有の文言。
  - `settings`: 設定画面（表示テーマ・API 認証・接続解除ダイアログ）。
  - `connection`: 接続バナー・認証情報フィールド（設定画面とオンボーディングで共用）。
  - `onboarding` / `devices` / `scenes` / `sensors` / `tray`: 各画面。
- **`errors`**: エラーコード → 文言（後述）。
- 判断基準: 2 画面以上で使う語彙は `common`（または共用機能 ns）、その画面だけの語彙は画面 namespace。

## エラーの多言語化（code → 文言）

Rust の `SwitchBotError` は `{ code, message, statusCode? }` にシリアライズされ、`invoke` の reject 値として渡る。フロントは **日本語メッセージを表示に使わず、安定コード（`code`）を翻訳キーにして表示端で文言化**する（言語切替に追従）。

- `src/i18n/error.ts`: Rust `ErrorCode`（serde camelCase）をミラーした `AppErrorCode` union（`unauthorized|rateLimited|apiStatus|offline|network|storage|missingCredentials|unknown`）と、`errorCodeOf(error)`（`AppError.code` を union 検証・未知は `unknown`）／`statusCodeOf(error)`（`apiStatus` の番号）。
- `src/data/ipc.ts`: `AppError` が `code` / `statusCode` / 診断用 `message`（英語フォールバック）を保持する。`message` は表示に使わずログ用途に限定する。
- ストア（`connection-store` / `device-store` / `notice-store`）は **文字列でなくコードを保持**する（`error: AppErrorCode | null`、`Notice = { id, code, statusCode? }`）。
- 表示端（`view-state` の `ErrorState` / `toast` / `settings-view` / `onboarding` / `scene-row` / `tray-popover`）が `useTranslation("errors")` の `t(code, { statusCode })` で描画する。
- `errors` namespace の各コード文言を ja/en で定義する。`apiStatus` は `{{statusCode}}` を補間し HTTP ステータス番号を保持する（`SwitchBotError` の任意フィールド `statusCode: Option<i64>` 由来）。

## Rust 由来の表示ラベルはフロントで文言化する

Rust `switchbot/mapping.rs` の DTO は **日本語ラベルを持たない**。フロントが安定 ID から翻訳する。

- カラー名: `ColorOptionDto.id`（`warm`/`neutral`/`cool`/`red`/`green`/`purple`）→ `devices:colorName.<id>`。
- センサー計測ラベル: `SensorMetricDto.icon`（`temperature`/`humidity`/`battery`/`motion`/`brightness`）→ `sensors:metric.<icon>`。
- センサー状態テキスト: `SensorMetricDto.state`（motion: `active`/`idle`、brightness: `bright`/`dim`）→ `sensors:state.<icon>.<state>`。
- 運転モード・風量・デバイス状態ラベルは `src/data/types.ts` の `airconModeLabel` / `airconFanLabel` / `deviceStatusLabel` が `devices` namespace の翻訳関数（`Translate`）を受け取り、キー選択と補間のみを担う。

## 翻訳対象外（意図的な例外）

- コメント（`//`・`/* */`）: 内部ドキュメント。
- `src/components/brand/*`: ブランド名 `SwitchBotler`・ワードマーク（`Switch` / `Botler`）。
- ユーザーが命名した実データ: デバイス名・シーン名・センサー `source`。
- 単位・記号（`%` / `℃` / `·` 等）: 言語非依存。

## native トレイメニューの言語伝搬

native 右クリックトレイメニュー（`src-tauri/src/lib.rs` の「ウィンドウを開く / 設定 / 終了」）は Rust が構築するため i18next で直接描画できない。フロントが解決した翻訳済みラベルを Tauri コマンドで Rust に渡し、Rust は保持した `MenuItem` の `set_text` で実行時更新する。

- **翻訳の SSoT**: i18n JSON に一本化する。Rust は翻訳表を持たず、受け取った文字列を反映するだけ（DRY）。使用キーは既存を再利用する（新規キーは追加しない）:
  - `tray:openWindow`（ja「ウィンドウを開く」/ en「Open window」）
  - `common:nav.settings`（ja「設定」/ en「Settings」）
  - `tray:quit`（ja「終了」/ en「Quit」）
- **Rust 側**: `setup_tray`（`lib.rs`）が生成した 3 つの `MenuItem` を `app.manage(TrayMenuItems { .. })` で managed state に保持する。コマンド `set_tray_menu_labels(open_window, settings, quit)`（`commands.rs`）が `State<TrayMenuItems>` から各項目を取り出し `set_text` する。invoke 引数は serde の camelCase 変換で `openWindow` → `open_window` に対応する。
- **フロント側**: `src/i18n/tray-native-menu.ts` の `syncNativeTrayMenu()` が翻訳済みラベルを組んで `invoke("set_tray_menu_labels", { openWindow, settings, quit })` する。`applyLanguage`（`language-store.ts`）から呼ぶため、起動ブートストラップ・`onRehydrateStorage`・`setLanguage` の全経路で追従し、言語変更時は再起動不要で即時反映される。IPC は外部境界のため失敗は握りつぶす（診断ログのみ）。
- **起動時フォールバック**: `setup_tray` の初期ラベルは `FALLBACK_LANGUAGE`（en）で構築する。native メニューは右クリックまで不可視（`show_menu_on_left_click(false)`）かつ Rust `setup()` は WebView ロード前に完了し、フロントが描画前の `applyLanguage` で即同期するため、初期フォールバックが露出する窓は不可視かつ極短で実害はない。
- **二重 invoke**: メイン / トレイ両ウィンドウが `applyLanguage` を実行するため同期が二重に走りうるが、`set_text` は同一ラベルで冪等のため無害。ウィンドウ判定は入れない（KISS）。

## ハードコード検出ガード

`scripts/check-i18n-hardcode.mjs` が `src` 配下の `.ts` / `.tsx` の **コメント以外**（JSX テキスト・文字列リテラル）に日本語（ひらがな・カタカナ・漢字）が残っていないか走査する。

- 実行: `mise run i18n-guard`（`mise run lint` にも含む）。
- 除外: `*.test.ts` / `*.test.tsx`・`src/i18n/locales/**`・`src/components/brand/**`。
- 組み込み: `lefthook`（pre-commit・`src` 全体を走査し過去の残存も検出）と CI（`.github/workflows/ci.yml` の frontend ジョブ）。

## キー命名規約

- ドット区切りの階層構造。リーフは lowerCamelCase（例: `settings.display.themeHint`、`language.system`）。
- UI 上の日本語そのものをキーにしない（意味を表す英語キーにする）。
- 同じ意味の語彙を複数キーに重複させない（DRY。共通語彙は `common` に集約）。

## 言語追加手順

1. `src/i18n/resolve.ts` の `SUPPORTED_LANGUAGES` に言語コードを追加する。
2. `src/i18n/locales/<lang>/` に各 namespace の JSON を追加する（既存キーを漏れなく揃える）。
3. `src/i18n/index.ts` の `resources` に登録する。
4. `src/components/language-select.tsx` の選択肢（`OPTIONS`）に追加し、`common.json` へラベルキー（`language.<lang>`）を足す。

## テスト方針

- 翻訳リソースの整合（各 namespace の ja/en キー集合一致・エラーコード網羅・補間 / 複数形 / 状態ラベル解決）は `src/i18n/i18n.test.ts` で検証する。テストは日本語ロケール既定（`src/test/setup.ts` が `navigator.language` を `ja` に固定し `changeLanguage("ja")`）で、en 検証は各テストで `changeLanguage("en")` する。
- 言語解決ロジック（`resolveLanguage`）と store 経由の言語即時反映を主対象に検証する（`src/i18n/resolve.test.ts` / `src/stores/language-store.test.ts`）。
- 画面の即時反映は store の `setLanguage` 経由で翻訳後テキストが再描画されることを assert する（`src/views/settings-view.test.tsx`）。radix Select の DOM 開閉操作テストは、外部ライブラリの責務であり `jsdom` で pointer 系のポリフィルを要するため、最小限に留める（`src/test/setup.ts` は現状維持）。
